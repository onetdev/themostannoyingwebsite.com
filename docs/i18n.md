# Internationalization (i18n)

The project supports UI and content translation.

Two different translation systems exist:

- UI translations
- content translations

## UI Translation

UI translation is implemented using **next-intl**.

All user-facing text must be translated. Hardcoded UI text should be avoided.

Translation files are located in:

```
apps/web/src/i18n/messages
```

or within feature modules:

```
src/features/*/i18n
```

## Translation File Patterns

Two translation patterns are used.

### Single File Pattern

Simple translations use a single file per locale.

Example:

```
en.ts
hu.ts
```

### Directory Pattern

Complex translations may use a directory structure:

```
en/
index.ts
variants.ts
```

- `index.ts` contains the main translation keys
- `variants.ts` contains large structured data sets

This pattern is used when translations include large datasets such as:

- comment templates
- lists of names
- structured content used by services

## Variants and Dynamic Imports

Large translation datasets can be dynamically imported by services.

This allows services to load only the data they need without loading the entire translation bundle.

## Content Translation

Content such as articles and media metadata is handled by the **content-api** package.

Location:

```
packages/content-api
```

This package bundles all content during the build process.

More details are available in:

```
packages/content-api/README.md
```

## Locale vs Language

The project distinguishes between:

- **language** → translation selection (`en`, `hu`)
- **locale** → formatting rules (`en-US`, `hu-HU`)

Translations typically operate on the **language level**, while formatting uses the **locale**.

Example:

```
language: hu
locale: hu-HU
```
